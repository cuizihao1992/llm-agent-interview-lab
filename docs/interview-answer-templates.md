# 面试回答模板

## 1. AI 记忆机制

回答结构：

1. 先指出不能全量塞 prompt。
2. 再讲分层记忆。
3. 然后讲生成时如何召回。
4. 最后讲异步更新和遗忘。

压缩版回答：

> 长期陪伴型 AI 的记忆不能简单依赖全量历史 prompt，因为成本、延迟和注意力干扰都会变严重。我会做分层记忆：近期多轮对话作为短期记忆直接保留；用户身份、偏好、经历这类事实抽取成结构化画像，用数据库精准检索；闲聊、感悟、项目背景这类非结构化内容切片后向量化，放到向量库做语义召回。生成回答时，把系统提示词、画像事实、召回片段和近期对话组合起来。记忆更新放到异步后台，包括事实抽取、冲突合并、入库和遗忘机制，保证体验和记忆质量。

## 2. RAG 数据工程

回答结构：

1. 先说 RAG 是数据工程问题。
2. 讲预处理和元数据。
3. 讲语义切片和父子块。
4. 讲向量化和索引。
5. 讲混合检索、重排和评估闭环。

压缩版回答：

> 我会把 RAG 向量数据建设看成完整的数据工程链路。首先对 PDF、网页、文档做解析清洗，统一格式，去掉乱码、广告、页眉页脚等噪音，同时提取时间、作者、权限、来源等元数据。切片上避免固定长度硬切，优先按标题、段落和语义边界切分，并保留 10% 到 20% 的重叠。可以采用子块检索、父块返回的方式兼顾召回精度和上下文完整性。向量化阶段通用场景用成熟 embedding 模型，专业领域要评估或微调。存储上千万级数据可以用 HNSW 索引。检索阶段用向量加关键词混合检索，再用 reranker 重排，最后用召回率、引用准确性、幻觉率、延迟和成本做闭环优化。

## 3. 长上下文和 RAG

回答结构：

1. 先给结论：不会替代，会融合。
2. RAG 解决广度，长上下文解决深度。
3. 解释成本、延迟、中间迷失、数据治理。
4. 给出 Long RAG 趋势。

压缩版回答：

> 长上下文模型不会直接取代 RAG，更可能是融合。RAG 负责从海量知识中做低成本、低延迟、可权限控制的粗召回，解决信息广度问题；长上下文模型负责对召回后的高质量材料做精读和跨文档推理，解决信息深度问题。即使模型上下文窗口足够长，全量输入仍然有 token 成本、响应延迟和中间迷失问题，也无法单独解决企业知识库里的权限、版本和引用治理。因此主流趋势是 Long RAG，用 RAG 先筛选高相关内容，再交给长上下文模型深度分析。

## 4. Transformer 长文本优化

回答结构：

1. 先讲 O(n^2) 注意力复杂度。
2. 再讲 KV Cache 显存和带宽。
3. 再讲位置编码外推。
4. 对应讲 FlashAttention、GQA、PagedAttention、RoPE Scaling、ALiBi。

压缩版回答：

> Transformer 处理超长上下文主要有三类瓶颈。第一是标准自注意力 O(n^2) 的计算复杂度，文本越长计算量平方级上涨。第二是推理时 KV Cache 带来的显存容量和带宽压力，高并发长文本下显存搬运会成为瓶颈。第三是位置编码外推性，模型在短文本训练后直接扩到超长文本，位置关系可能泛化不好。优化上，FlashAttention 通过算子融合和分块计算减少显存 IO；GQA 让多组 Query 共享 Key 和 Value，减少 KV Cache；PagedAttention 用分页方式管理 KV Cache，提高显存利用率和并发能力；RoPE Scaling、ALiBi 等方法提升长文本位置建模能力。整体是从算子、架构、显存和位置编码多个层面组合优化。
