# RAG 向量数据全流程工程化方案

## 面试题

如何构建一个高质量的 RAG 向量数据工程链路？

## 核心判断

RAG 的效果不只取决于大模型，数据质量决定系统下限，工程链路决定系统上限。

一个完整 RAG 链路通常包括：

```text
原始数据 -> 解析清洗 -> 元数据提取 -> 语义切片 -> 向量化 -> 向量库索引 -> 检索 -> 重排 -> 生成回答 -> 评估反馈
```

## 1. 数据预处理

原始数据可能来自 PDF、网页、Word、Markdown、数据库、知识平台和聊天记录。

预处理目标：

- 格式统一。
- 去除乱码、广告、导航栏、页眉页脚等噪音。
- 保留标题、段落、表格、列表等结构。
- 提取元数据。

常见元数据：

- 文档标题。
- 作者。
- 创建时间和更新时间。
- 来源 URL。
- 权限范围。
- 业务分类。
- 文档版本。

元数据非常重要，因为后续检索不仅靠语义相似度，还经常需要权限过滤、时间过滤和业务域过滤。

## 2. 文本切片

固定长度切割容易破坏语义边界。更推荐语义切分：

- 按标题、段落、句子、列表层级切。
- 保留 10% 到 20% 的滑动窗口重叠。
- 对长文档使用父子块架构。

父子块架构：

- 子块用于向量检索，粒度小，召回更精准。
- 父块用于返回给模型，保留更完整上下文。

这样可以兼顾检索精度和回答所需的上下文完整性。

## 3. 向量化

通用场景可以使用主流 Embedding 模型。

专业领域需要重点关注：

- 领域术语是否能被正确编码。
- 中英文混合语料表现。
- 长文本截断策略。
- 是否需要用领域数据微调 embedding 模型。

医疗、法律、金融等专业场景通常需要评估或微调专属模型。

## 4. 向量库存储

千万级数据常见方案是 HNSW 索引，用于平衡速度和召回精度。

工程上还要考虑：

- 批量写入。
- 增量更新。
- 删除和重建索引。
- 多租户隔离。
- 元数据过滤。
- 备份和恢复。

## 5. 检索闭环优化

单纯向量召回不够稳定，常见优化组合是：

- 向量检索：解决语义相似。
- 关键词检索：解决专有名词、编号、精确匹配。
- 混合检索：融合向量分数和关键词分数。
- 重排序：用 reranker 对候选片段重新打分。
- 查询改写：把模糊问题改写成更适合检索的问题。
- 模拟答案检索：先生成可能答案，再用答案去检索相关证据。

## 6. 评估指标

RAG 不应只看最终回答，还要拆开评估：

- 召回率：相关文档是否被召回。
- 精确率：召回内容是否足够相关。
- 引用准确性：答案是否能被证据支持。
- 幻觉率：模型是否编造。
- 延迟：检索和生成总耗时。
- 成本：embedding、向量库、大模型 token 成本。

## 面试回答模板

> 我会把 RAG 看成数据工程问题，而不是只接一个向量库。首先对 PDF、网页、文档做解析和清洗，统一格式，去掉噪音，并提取时间、作者、权限等元数据。切片上避免固定长度硬切，优先按语义结构切分，并设置 10% 到 20% 的重叠。为了兼顾精度和上下文完整性，可以用子块检索、父块返回。向量化阶段通用场景用成熟 embedding 模型，专业领域需要评估或微调。向量库在千万级数据下可以用 HNSW 索引，同时做批量入库和元数据过滤。检索时采用向量加关键词的混合检索，再用 reranker 重排，最后通过召回率、引用准确性、幻觉率、延迟和成本形成持续优化闭环。
