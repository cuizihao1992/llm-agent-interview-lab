# 长期陪伴型 AI 角色的记忆机制设计

## 面试题

长期陪伴型 AI 角色对话轮次越来越多，如何设计记忆机制？

## 核心矛盾

不能把全部历史对话都塞进提示词。

原因有三个：

- 成本高：输入 token 越多，推理成本越高。
- 延迟高：上下文越长，推理和网络传输越慢。
- 注意力分散：模型容易被无关历史干扰，甚至遗漏真正重要的信息。

核心方案是分层管理记忆，把不同类型的信息放进不同存储，并在生成回答前按需召回。

## 三类记忆

### 1. 短期记忆

短期记忆保存最近几轮原始对话，用于保障当前对话的上下文连贯性。

特点：

- 时间窗口短。
- 保留原始表达。
- 通常直接拼入 prompt。

适合存储：

- 当前任务进展。
- 最近的问题和回答。
- 刚刚提到但还没有沉淀为长期事实的信息。

### 2. 实体画像记忆

实体画像记忆用于保存事实型、稳定型信息。

例如：

- 用户姓名、职业、所在城市。
- 用户偏好。
- 用户长期目标。
- 用户经历和重要关系。

这类信息适合做结构化存储，比如关系型数据库、文档数据库或 KV 存储。

示例结构：

```json
{
  "user_id": "u_001",
  "facts": [
    {
      "key": "career_goal",
      "value": "准备大模型 Agent 算法岗位面试",
      "confidence": 0.92,
      "evidence": "用户多次询问 Agent 和 RAG 面试题",
      "updated_at": "2026-06-18"
    }
  ]
}
```

优势：

- 查询精准。
- 易更新、易冲突处理。
- 适合做权限控制和审计。

### 3. 长期情景记忆

长期情景记忆用于保存非结构化、语义型内容。

例如：

- 闲聊中的情绪表达。
- 某次项目讨论的背景。
- 用户曾经表达过的困惑或偏好语境。

这类内容适合切片后向量化，存入向量数据库，用语义相似度召回。

## 调用流程

一次回答通常拼接以下内容：

1. 系统提示词。
2. 用户画像和关键事实。
3. 语义召回的长期情景片段。
4. 最近多轮短期对话。
5. 当前用户问题。

## 后台更新机制

为了不阻塞用户体验，记忆更新应采用异步流程。

主流程：

1. 用户发起消息。
2. AI 先基于已有记忆生成回复。
3. 后台任务抽取新事实和情景片段。
4. 更新结构化画像和向量库。
5. 做冲突清理、合并和遗忘。

## 遗忘机制

长期陪伴型 AI 不能只积累不删除。记忆需要动态维护：

- 过期事实降权或删除。
- 低价值闲聊片段降低召回权重。
- 与新事实冲突的旧事实进入待确认状态。
- 敏感信息按权限和用户设置清理。

## 面试回答模板

> 我不会把全部历史直接塞进 prompt，而是做分层记忆。近期几轮对话作为短期记忆直接拼接，用户偏好、身份、经历这类事实抽取成结构化画像，用数据库精准查询；感悟、闲聊、项目背景这类非结构化内容切片后向量化，放入向量库做语义召回。生成回答时，把系统提示词、画像事实、召回片段和近期对话组合起来。记忆更新放到异步后台做，包括事实抽取、入库、冲突合并和遗忘更新，这样既保证体验，也能让记忆持续迭代。
